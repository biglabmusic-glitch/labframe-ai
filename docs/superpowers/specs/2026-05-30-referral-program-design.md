# Реферальная программа «Приведи друга» — дизайн

Дата: 2026-05-30
Статус: одобрен (брейншторм), готов к плану реализации

## Цель

Дать пользователям приглашать друзей по реферальной ссылке **и** по промокоду.
За первую оплату приглашённого друга обе стороны получают бонусные генерации.

## Ключевые решения

- Механизм: **и ссылка, и промокод** (ссылка для шеринга в один тап, код — для офлайна).
- Награда: **+генерации обоим** (двусторонняя), копится на будущее.
- Условие начисления: **первая оплата приглашённого друга** (не вход, не первая генерация).
  Это главный анти-абьюз: фейковому аккаунту пришлось бы реально заплатить.
- Размер награды: **настраивается через env**, старт `REFERRER_BONUS=10`, `REFEREE_BONUS=5`.

## Контекст проекта

- Лимиты генераций сейчас отключены: `app/src/lib/feature-flags.ts` → `LIMITS_DISABLED = true`,
  БД-триггер `enforce_usage_limit` снят миграцией `0005_disable_usage_limit.sql`.
- Поэтому бонус «+N генераций» (рост `usage_limit`) сейчас не даёт мгновенного эффекта юзеру —
  он **копится** и заработает, когда включим лимиты/оплату. В UI формулировки нейтральные
  («начислится / копится»), без обещания мгновенного роста лимита.
- Оплаты пока нет (Telegram Stars / ЮKassa / Продамус — в планах). Поэтому фича разбита на
  два слоя: привязки копятся с первого дня, а начисление награды «выстреливает» из будущего
  платёжного вебхука.

## Модель данных (миграция 0010_referrals.sql)

### Изменения в `users`
- `ref_code text unique` — короткий код вида `ZUB-AB12`. Генерится при первом `/me`, если пусто.
- `referred_by bigint references users(id)` — кто пригласил этого юзера. Пишется ОДИН раз.
- `referral_rewarded boolean not null default false` — был ли уже выдан бонус за этого юзера
  (страховка от двойного начисления на уровне приглашённого).

### Новая таблица `referrals` (журнал привязок, один ряд = один приглашённый)
- `id` — pk.
- `referrer_id bigint not null references users(id)` — кто позвал.
- `referee_id bigint not null unique references users(id)` — кого позвали.
  UNIQUE физически запрещает привязать одного друга к двум пригласившим.
- `status text not null default 'joined'` — `'joined' | 'paid'`.
- `created_at timestamptz not null default now()`.
- `rewarded_at timestamptz` — когда начислили награду.
- Индексы: `(referrer_id)`, `(status)`.

Обоснование разделения: `users.referred_by` — быстрый ответ «кто меня привёл»;
таблица `referrals` — источник для счётчиков «скольких привёл / сколько оплатили».

## Поток привязки

1. Пригласивший шарит `t.me/labframe_ai_bot/app?startapp=ref_ZUB-AB12` или диктует `ZUB-AB12`.
2. Друг открывает Mini App. Фронт читает `WebApp.initDataUnsafe.start_param`
   (сейчас в `app/src/telegram/webapp.ts` не читается — добавим) или принимает ручной ввод кода.
3. Бэк ставит привязку только при соблюдении ВСЕХ анти-абьюз правил.

## Анти-абьюз (всё проверяется на бэке, не на фронте)

| Правило | Зачем |
|---|---|
| Привязка только новому юзеру (в окне `REFERRAL_NEW_USER_WINDOW_MIN` минут от created_at и/или без job) | Нельзя привязать старого юзера задним числом |
| `referred_by` пишется один раз и не меняется | Нельзя перепривязаться к более щедрому коду |
| Свой код запрещён (referrer_id ≠ referee_id) | Самоприглашение |
| `referee_id UNIQUE` в БД | Один друг — один пригласивший навсегда |
| Код должен существовать (валидный referrer) | Мусорные коды отсекаются |
| Награда только при `status='paid'` и переходе ровно одного ряда, атомарно | Двойное начисление невозможно |

**Осознанно НЕ защищаем:** создание фейковых TG-аккаунтов и самоприглашение. Это обезврежено
тем, что награда завязана на реальную оплату друга — накрутка дороже выгоды.

## API (Edge Functions)

### `/me` (изменяем) — `supabase/functions/me/index.ts`
- После upsert: если `ref_code` пуст — сгенерировать уникальный (`ZUB-` + 4 символа из `crypto`,
  проверка коллизии перегенерацией).
- Вернуть фронту: `refCode`, `referralsCount`, `referralsPaid`, `referralBonusEarned`.

### `/apply-referral` (новая) — `POST { code?: string, startParam?: string }`
- Авторизация по initData.
- Источник кода: `code` (ручной ввод) или `startParam` (`ref_ZUB-AB12` → `ZUB-AB12`).
- Прогон всех анти-абьюз правил. Нарушение → мягкий `{ ok:false, reason }` (не 500).
- Успех → `users.referred_by` + ряд `referrals` со `status='joined'`.
- Идемпотентна: повторный вызов при стоящей привязке → `{ ok:true, already:true }`.

### `_shared/referral.ts` (новый) — `grantReferralReward(refereeId)`
- Вызывается в момент первой оплаты друга (из будущего платёжного вебхука).
- Атомарно: `referrals` где `referee_id=refereeId, status='joined'` → `status='paid'`, `rewarded_at=now()`.
- Только если переход реально произошёл (обновлён ровно один ряд):
  - пригласившему `usage_limit += REFERRER_BONUS`;
  - другу `usage_limit += REFEREE_BONUS`;
  - `users.referral_rewarded = true` для друга.
- Двойного начисления нет за счёт условия атомарного перехода.

### Точка оплаты (временно, пока нет вебхука)
- Добавить в админку (`supabase/functions/admin/index.ts`) action `mark-paid` (userId) для теста
  начисления / имитации первой оплаты.
- Когда подключим ЮKassa/Продамус — вебхук вызовет тот же `grantReferralReward`, админ-экшн можно убрать.

### Env (с дефолтами)
- `REFERRAL_REWARD_REQUIRES_PAYMENT=true`
- `REFERRER_BONUS=10`
- `REFEREE_BONUS=5`
- `REFERRAL_NEW_USER_WINDOW_MIN=60`

## Фронт

### Новый экран `ScreenInvite.tsx` (роут `invite`)
- Объяснение: «Друг оплачивает план → ты +10, друг +5».
- Твоя ссылка `t.me/labframe_ai_bot/app?startapp=ref_<CODE>` + «Поделиться» + «Скопировать».
- Твой промокод `ZUB-AB12` крупно + «Скопировать».
- Статистика: «Приглашено: N · Оплатили: M · Получено генераций: K» (из `/me`).
- Опц.: список приглашённых (имя/username + статус Зашёл/Оплатил).

### Точка входа
- Карточка/кнопка «Пригласить друга» на `ScreenHome` (и/или `ScreenMyPlan`).

### Приём входящей привязки (для друга)
- В `app/src/telegram/webapp.ts` начать читать `WebApp.initDataUnsafe.start_param`.
- После `/me` при старте: если `start_param` вида `ref_*` и юзер новый — тихо вызвать `/apply-referral`,
  показать мягкий тост «Промокод применён, +5 генераций после первой оплаты».
- На экране «Пригласи друга» — поле ручного ввода кода → тот же `/apply-referral`.

### Учёт LIMITS_DISABLED
- Пока `LIMITS_DISABLED = true`: бонус показываем как «начислится / копится», без обещания
  мгновенного роста лимита.

## Безопасность (наследуем текущие практики)
- Все эндпоинты — за `authorize(req)` (Telegram initData HMAC).
- Запись только через service-role в Edge Functions; RLS default-deny остаётся.
- Никаких секретов/диагностики в теле ответов (согласуется с недавним фиксом auth.ts).
- Валидация кода: вырезать лишние символы, сравнивать по нормализованному виду (upper-case, trim).

## Вне скоупа (YAGNI)
- Многоуровневые рефералы (рефералы рефералов).
- Денежные выплаты/вывод средств.
- Срок жизни/истечение реф-кодов.
- Реальный платёжный вебхук (отдельный спек, когда выберем провайдера).
